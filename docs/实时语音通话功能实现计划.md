LFBot × Linyu 实时语音通话功能实现计划
Context
LFBot 是一个 AI 伴侣项目，已通过 LinyuAdapter 接入自建 IM 工具 linyu，实现了文本/图片/语音消息的交互。现在需要实现实时语音通话功能——用户在 linyu 客户端点击"语音通话"按钮，就能像和真人打电话一样与 AI 进行实时语音对话。

核心挑战: linyu 的语音通话基于 WebRTC P2P 模式（音频流直接在两个客户端之间传输），但 AI 没有浏览器/客户端。解决方案是用 aiortc（Python WebRTC 库）在服务端模拟一个 WebRTC 端点，对 linyu 完全透明。

技术选型:

音频传输: aiortc 模拟 WebRTC 端点
AI 理解: 千问 Omni Realtime（音频输入，文本输出）
语音合成: 千问 TTS Realtime（文本输入，音频输出）
配置管理: 管理后台前端可视化配置（避免手改 config.yaml）
集成方式: 集成到现有 LFBot 进程中（非独立微服务）
对 linyu 零侵入: AI 完全模拟 linyu 用户行为
第一版支持双向: 用户拨 AI + AI 主动拨用户

核心数据流:


用户说话 → linyu WebRTC → aiortc 收音频(48kHz PCM)
  → 降采样(48kHz→16kHz) → 千问 Omni Realtime(音频入)
  → AI 文本增量(response.text.delta)
  → 文本分句/流式切片 → 千问 TTS Realtime(文本入)
  → TTS 音频(优先 48kHz PCM) → aiortc 发音频 → linyu WebRTC → 用户听到

2.2 通话状态机 (session.py)

用户拨打 AI:
  IDLE → INCOMING_RINGING (收到 invite)
       → CONNECTING (发送 accept, 开始 WebRTC 握手)
       → CONNECTED (ICE 连接建立, 音频流通)
       → ENDED (任一方 hangup)

AI 拨打用户:
  IDLE → OUTGOING_RINGING (发送 invite)
       → CONNECTING (收到 accept, 开始 WebRTC 握手)
       → CONNECTED (ICE 连接建立, 音频流通)
       → ENDED (任一方 hangup)
状态枚举: IDLE / INCOMING_RINGING / OUTGOING_RINGING / CONNECTING / CONNECTED / ENDED / REJECTED / CANCELLED / FAILED

终态说明:

REJECTED — 对方明确拒接或忙线拒绝
CANCELLED — 主叫在接通前主动取消
FAILED — 握手/媒体/上游依赖异常导致失败

CallSession 持有:

call_id (uuid)
peer_user_id (对方 linyu 用户 ID)
is_outgoing (是否 AI 主动拨出)
state (CallState)
webrtc_endpoint (WebRTCEndpoint 实例)
realtime_client (QwenOmniClient 实例)
tts_client (QwenTTSClient 实例)
audio_pipeline (AudioPipeline 实例)
时间戳 (created_at, connected_at, ended_at)
2.3 VoiceCallManager (manager.py)
职责: 通话生命周期管理，与 LinyuAdapter 集成的桥梁。

关键方法:

handle_video_signal(content) — 处理来自 WebSocket 的 video 类型信令
initiate_call(target_user_id) — AI 主动拨打用户
_handle_invite() — 处理来电：访问控制检查 → 模拟接听延迟(1~3s, 拟人化) → accept → 初始化资源
_handle_accept() — 对方接听 AI 拨号：初始化资源 → 创建 Offer → 发送
_handle_offer() — 处理 WebRTC Offer：aiortc 生成 Answer → 发送
_handle_answer() — 处理 WebRTC Answer：设置远端描述
_handle_candidate() — 处理 ICE Candidate：添加到 aiortc
_handle_hangup() — 处理挂断：清理资源
_setup_call_resources(session) — 初始化 WebRTC + Omni Realtime + TTS Realtime + AudioPipeline
_end_call(session, reason) — 结束通话，清理所有资源，保存通话记录到记忆
_ring_timeout(session, timeout=30) — 拨号超时自动挂断
与 LinyuAdapter 的集成:

持有 adapter 引用，复用 adapter._request_json() 调用 linyu HTTP API
复用 adapter._check_user_access() 做访问控制
复用 adapter 已有的认证 token
2.4 WebRTC 端点 (webrtc_endpoint.py)
基于 aiortc 实现，模拟 linyu 客户端的 WebRTC 行为。

关键设计:

RTCPeerConnection 配置 ICE 服务器（与 linyu 客户端一致的 STUN/TURN）
自定义 AudioOutputTrack(MediaStreamTrack) — 用于向用户发送 AI 语音
内部维护 asyncio.Queue，recv() 方法从队列取 PCM 数据构造 av.AudioFrame
无数据时发送静音帧，保持流畅
监听远端 track 事件 — 收到用户音频后通过回调传给 AudioPipeline
监听 icecandidate 事件 — 收集本地 ICE candidates 通过回调发送给 linyu
监听 connectionstatechange — 通知上层连接状态变化
关键方法:

handle_offer(sdp) -> answer_sdp — 被叫方：处理 Offer 生成 Answer
create_offer() -> offer_sdp — 主叫方：创建 Offer
handle_answer(sdp) — 主叫方：处理 Answer
add_ice_candidate(candidate_data) — 添加远端 ICE Candidate
send_audio_frame(pcm_data, sample_rate) — 推送 AI 音频到发送队列
close() — 关闭连接
ICE Candidate 处理:

aiortc 的 onicecandidate 回调触发时，通过 on_local_candidate 回调通知 VoiceCallManager
VoiceCallManager 调用 adapter._request_json("POST", "/v1/api/video/candidate", ...) 发送给 linyu
2.5 千问 Omni Realtime 客户端 (realtime_client.py)
通过 WebSocket 连接千问 Omni Realtime API，实现“音频输入、文本输出”。

连接参数:

认证: Authorization: Bearer {api_key}
模型: qwen3-omni-flash-realtime (或配置指定)
会话配置 (session.update):


{
  "modalities": ["text"],
  "instructions": "角色设定 system prompt",
  "input_audio_format": "pcm16",
  "input_audio_transcription": { "model": "qwen3-asr-flash-realtime" },

    "type": "server_vad",
    "threshold": 0.5,
    "prefix_padding_ms": 300,
    "silence_duration_ms": 500
  }
}
关键事件处理:

response.text.delta — AI 回复文本增量 → 传给 TTS 客户端流式合成，并用于记忆系统记录
input_audio_buffer.speech_started — 用户开始说话 → 触发打断，发送 response.cancel，并清空 TTS 播放队列
input_audio_buffer.speech_stopped — 用户停止说话
conversation.item.input_audio_transcription.completed — 用户语音转录完成 → 日志/记忆
response.done — AI 一轮回复完成
error — 错误处理
关键方法:

connect() — 建立 WebSocket 连接，发送 session.update
send_audio(pcm_data: bytes) — 发送音频到 Omni Realtime API (input_audio_buffer.append)
update_instructions(prompt: str) — 更新系统提示词（注入记忆上下文）
close() — 关闭连接

2.6 千问 TTS Realtime 客户端 (tts_client.py)
通过 WebSocket 连接千问 TTS Realtime API，实现“文本输入、音频输出”。

关键事件处理:

response.audio.delta — TTS 音频增量 → 通过回调传给 WebRTCEndpoint 发送
response.done — 一轮合成完成
error — 错误处理

关键方法:

connect() — 建立 WebSocket 连接，发送 session.update（voice/sample_rate 等）
send_text_delta(text: str) — 发送文本增量进行流式合成
flush() — 提交当前分句缓冲，触发尽快出声
clear() — 打断时清空当前合成与播放队列
close() — 关闭连接

2.7 音频处理管道 (audio_pipeline.py)
职责: WebRTC 音频格式与 Omni/TTS 音频格式之间的转换。

格式对照:

环节	采样率	格式
WebRTC (aiortc) 输入/输出	48kHz	16-bit PCM, mono
千问 Omni Realtime 输入	16kHz	16-bit PCM, mono
千问 TTS Realtime 输出	48kHz(优先，可配置)	16-bit PCM, mono
处理链路:


用户音频: WebRTC 48kHz → 降采样 → 16kHz → 千问 Omni Realtime
AI 音频:  千问 TTS 48kHz(优先) → 直接送 WebRTC（若非 48kHz 再重采样）
实现: 使用 soxr 库的流式重采样器 (soxr.ResampleStream)，保持 chunk 边界状态，避免音频断裂。降级方案统一使用 numpy 线性插值（不额外引入 scipy 依赖）。

# 现有代码只处理 type="msg"，需要新增:
async def _process_ws_message(self, data: Dict[str, Any]):
    msg_type = data.get("type")
    if msg_type == "msg":
        content = data.get("content")
        if not isinstance(content, dict):
            return
        source = content.get("source")
        if source == "group":
            await self._handle_group_message(content)
        else:
            await self._handle_private_message(content)

        content = data.get("content")                  # ← 新增

四、关键技术风险与应对
风险	影响	应对
aiortc 与 linyu 客户端 WebRTC 兼容性	ICE 连接失败	先用最简配置测试，逐步调整 SDP/ICE 参数；确保 STUN/TURN 配置一致
Omni/TTS 双链路增加时延	首包变慢、打断体感变差	使用 response.text.delta 增量立刻喂给 TTS；开启 48kHz 直出减少下行重采样
文本分句策略不当	语音断句生硬	按标点+最大字数做流式切片，句末优先提交，必要时补短停顿
aiortc 在 Windows 上的兼容性	安装或运行失败	aiortc 依赖 libavcodec，Windows 需要预编译 wheel；备选方案用 WSL
音频重采样质量	听感差	使用 soxr 流式重采样器（高质量）；降级用 numpy 线性插值
linyu WebSocket 消息中 video 类型的具体格式	信令解析失败	已从源码确认格式：{"type":"video","content":{"type":"offer",...}}，与分析一致
长通话稳定性	内存泄漏/连接断开	设置最大通话时长；定期检查连接状态；资源显式回收
